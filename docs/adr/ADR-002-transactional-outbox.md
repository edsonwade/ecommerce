# ADR-002: Transactional Outbox Pattern for Reliable Event Publishing

**Date:** 2026-04-11
**Status:** Accepted
**Author:** Vanilson

---

## Context

`order-service` must publish a Kafka event (`order.requested`) every time a new order is persisted. This creates the **dual-write problem**: two separate, non-atomic writes are required вҖ” one to PostgreSQL and one to the Kafka broker.

Without a coordination strategy, two failure modes are possible:

1. The database write succeeds, but the Kafka publish fails вҶ’ the saga never starts, yet the order exists in the database. The customer's order is silently stuck in `PENDING` forever.
2. The Kafka publish succeeds, but the database write fails (or is rolled back) вҶ’ downstream services process an order that does not exist, leading to phantom inventory reservations and payments.

A distributed transaction (XA/2PC) between PostgreSQL and Kafka is theoretically possible but was ruled out (see ADR-001 for reasons). A purely in-memory "publish after commit" approach cannot survive application crashes between the commit and the publish call.

---

## Decision

We implemented the **Transactional Outbox pattern** in `order-service`.

When an order is created, the service writes two records within a **single database transaction**:

1. The `Order` entity вҖ” the business record.
2. An `OutboxEvent` entity вҖ” a serialised representation of the Kafka message to be sent.

Because both writes share the same ACID transaction, they are guaranteed to either both succeed or both be rolled back. The Kafka broker is never written to inside this transaction.

A separate scheduled component, `OutboxEventPublisher`, polls the `outbox_event` table every 5 seconds for records in `PENDING` status and publishes them to Kafka. On successful delivery (confirmed by Kafka's `SendResult` callback), the record is updated to `PUBLISHED`. On failure, `retryCount` is incremented. After 5 failed attempts, the record is marked `FAILED` and requires manual intervention.

The `partitionKey` on each `OutboxEvent` is set to `correlationId` (the order saga ID), ensuring all events for a given order are routed to the same Kafka partition, preserving causal ordering.

`OutboxEvent.id` is a UUID generated by the database. The separate `eventId` column carries the application-level UUID that matches the Kafka event payload вҖ” this is what downstream consumers use for **idempotency deduplication**. The two identifiers are intentionally distinct: `id` identifies the outbox row; `eventId` identifies the Kafka message.

## Alternatives Considered

### Publish directly to Kafka after `@Transactional` commit

Call `kafkaTemplate.send(...)` immediately after `orderRepository.save(...)`, inside the same service method.

**Why rejected:**
- Spring's `@Transactional` commit and the subsequent Kafka send are two separate operations. A JVM crash or network failure between them leaves the order saved but the event never published.
- Even wrapping the publish in a `TransactionSynchronizationAdapter.afterCommit` callback doesn't eliminate the gap вҖ” if the process dies in that callback, the event is lost permanently.

### Debezium Change Data Capture (CDC)

A Debezium connector tails the PostgreSQL Write-Ahead Log (WAL) and publishes `outbox_event` row inserts directly to Kafka with near-zero latency.

**Why not adopted yet:**
- Requires a Debezium connector cluster as additional infrastructure, increasing operational complexity.
- The current 5-second polling delay is acceptable for Phase 3 of this project.
- The codebase already notes this as a Phase 4 migration path (`OutboxEventPublisher` Javadoc).

### Kafka Transactions (Exactly-Once Semantics)

Use Kafka's transactional producer API with `isolation.level=read_committed` on consumers to achieve exactly-once delivery.

**Why rejected:**
- Kafka transactions do not span a relational database. They only guarantee atomicity *within* Kafka (producer writes to multiple topics atomically). The fundamental dual-write problem between PostgreSQL and Kafka remains unsolved.
- Significantly more complex to configure correctly; higher risk of producer fencing bugs under failure scenarios.

---

## Consequences

**Positive:**
- The business transaction (order creation) and the intent to publish an event are **atomically coupled**. An order can never exist without its corresponding outbox record.
- At-least-once delivery is guaranteed. Even if the application crashes and restarts, `OutboxEventPublisher` will find the `PENDING` record and retry.
- The outbox table provides a durable audit log of every event ever published and its delivery status.
- `FAILED` events are visible to operations teams and can be replayed manually.

**Negative / Trade-offs:**
- There is an inherent **up-to-5-second latency** between order creation and the `order.requested` Kafka event being published. This is acceptable because the client receives `202 Accepted` immediately and polls for status.
- The `outbox_event` table grows over time. A periodic cleanup job should archive or delete `PUBLISHED` records older than a retention window (not yet implemented).
- `OutboxEventPublisher` runs as a single-threaded scheduler. Under very high load, the 5-second poll interval may introduce a backlog. Mitigation: switch to Debezium CDC (Phase 4).
- Consumers **must** implement idempotency checks using `eventId`, because at-least-once delivery means duplicates are possible if the outbox publisher crashes after a successful Kafka send but before updating the status to `PUBLISHED`.
